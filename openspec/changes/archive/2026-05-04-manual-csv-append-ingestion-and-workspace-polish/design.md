## Context

The existing pipeline discovers CSV files from dataset folders, registers `source_files`, writes Raw rows with source lineage, and builds Normalized/Silver data with deterministic upserts and incremental state support. Raw ingestion currently deduplicates within the `source_file_id + raw_row_num` boundary, while normalized and Silver builders deduplicate by business keys and conflict fields.

The requested manual upload flow must not become a second pipeline. It should be a controlled intake surface that feeds the same ingestion and normalization machinery with a single selected file scope. The user chooses the semantic dataset because file names can be wrong or inconsistent.

## Investigation Findings

### Reusable raw-ingest seam
- `scripts/ingest_raw.py` already contains the reusable primitives for single-file ingestion: `count_csv_rows`, `upsert_source_file`, `create_run`, `create_batch`, `count_raw_rows_for_source_file`, `build_raw_ingest_metrics`, `persist_raw_ingest_failure`, and `ingest_file`.
- `ingest_file` is already file-scoped and writes lineage through `SourceFile`, `IngestionBatch`, `PipelineRun`, and `PipelineRunStep`.
- The remaining gap is orchestration: `main()` still discovers files from the dataset root and loops over historical files. There is no manual preflight, staging token, or controlled intake directory yet.

### Reusable normalized/Silver seam
- `scripts/build_normalized.py` already exposes `create_normalized_run`, `run_dataset_preflight_quality_audit`, `resolve_start_after_id`, `process_licitaciones`, `process_ordenes_compra`, `dedupe_rows`, `upsert_rows`, `flush_if_needed`, and `flush_remaining`.
- Normalized and Silver writes already use business-key conflict handling rather than raw file identity.
- The missing seam is a scoped boundary for one staged uploaded file. Current incremental behavior is dataset-wide and raw-id driven (`start_after_id` / `last_processed_raw_id`), so one uploaded file still needs a first-class source-file scope before we can promise bounded downstream processing.

### Current lineage and status coverage
- `backend/models/operational.py` already has enough operational columns for the manual flow to report and persist status without a new table in the first pass:
  - `SourceFile.source_meta`
  - `PipelineRun.config` and `PipelineRun.error_summary`
  - `PipelineRunStep.rows_in`, `rows_out`, and `rows_rejected`
  - `IngestionBatch.total_rows`, `loaded_rows`, and `rejected_rows`
- `backend/api/routers/operations.py` currently exposes read-only runs/files/summary endpoints; there is no manual upload router yet.

### Duplicate behavior
- Raw intentionally retains renamed-file lineage because its uniqueness boundary is `source_file_id + raw_row_num` in `backend/models/raw.py`.
- Normalized and Silver collapse by business key or conflict key through `dedupe_rows` plus `on_conflict_do_update` / `on_conflict_do_nothing` in `scripts/build_normalized.py`.
- Concrete conflict keys are already explicit in the script:
  - licitaciones: `codigo_externo`, `codigo_externo + codigo_item`, `oferta_key_sha256`
  - ordenes de compra: `codigo_oc`, `codigo_oc + id_item`
  - Silver: `notice_id`, `notice_id + item_code`, `purchase_order_id`, `purchase_order_id + line_item_id`, and related relation keys
- Test-plan implication: a renamed file with repeated business rows should produce a new raw lineage record while normalized and Silver should surface skipped/existing counts instead of new canonical duplicates.

### Contract freeze
- Dataset enum values remain `licitacion` and `orden_compra`.
- UI labels remain `Licitaciones` and `Ordenes de compra` in the workspace contract.
- Uploaded filenames are metadata only and must not infer the dataset.
- The draft upload API shape in this change remains `preflight`, `process`, and `jobs/{job_id}` under `/uploads/procurement-csv/...`.

Concrete motivating example:
- `C:\Users\nenri\Downloads\2026-4 (1)\lic_2026-4.csv`
- The file is a licitaciones CSV even though its name and folder do not match the original bulk-load discovery layout.
- The header is semicolon-delimited and includes the licitacion required columns (`Codigo`, `CodigoExterno`, `Tipo de Adquisicion`, `FechaPublicacion`, `FechaCierre`, `Codigoitem`).
- The manual flow should accept this file only because the operator explicitly selects `Licitaciones`, not because the filename starts with `lic_`.
- The staged canonical name should be generated by the system, while the original filename remains audit metadata.

Second motivating example:

- The file is an órdenes de compra CSV even though the filename does not identify the dataset and does not match the bulk-load `orden-compra/*.csv` or `orden_compra/*.csv` discovery conventions.
- The header is semicolon-delimited and includes the orden-compra required columns (`Codigo`, `FechaEnvio`, `Estado`, `DescripcionTipoOC`, `IDItem`, `codigoProductoONU`, `totalLineaNeto`).

## Goals / Non-Goals

**Goals**
- Let an operator upload one CSV and explicitly choose `licitacion` or `orden_compra`.
- Validate extension, content type, required columns, row count, and file hash before processing.
- Preserve source lineage from uploaded file through raw, normalized, and Silver outputs.
- Process only the uploaded file's dataset scope and downstream affected raw rows by default.
- Skip duplicates through existing idempotent semantics and report duplicate/skipped counts.
- Keep the full historical pipeline untouched unless the operator explicitly runs the existing full command.
- Show upload progress, preflight, errors, and result telemetry in the UI.
- Improve `/licitaciones` header, KPI cards, pulse metrics, and expanded evidence section visual quality.

**Non-Goals**
- Rebuild the whole ingestion architecture.
- Add predictive scoring, AI recommendations, assignment, notes, or editable opportunity workflow.
- Trust the uploaded filename as the dataset selector.
- Process both datasets from a single manual upload unless the user explicitly uploads separate files.
- Rerun all raw, normalized, or Silver data by default.
- Persist arbitrary uploaded file paths or allow path traversal.
- Add a public unauthenticated production upload surface without future auth/authorization review.

## Decisions

1. **Manual dataset selector is authoritative.**
   - Rationale: filenames can be wrong; operator intent must determine `licitacion` vs `orden_compra`.

2. **Use controlled server-side intake storage.**
   - Rationale: uploaded file bytes must land in a known folder, with sanitized canonical names, hash metadata, and no user-controlled path writes.

3. **Preflight before write.**
   - Rationale: malformed CSV, wrong delimiter, missing required columns, oversized files, empty files, or wrong dataset contracts should fail before raw/normalized data changes.

4. **Reuse existing pipeline code paths.**
   - Rationale: lineage, telemetry, idempotency, and data contracts already exist; duplicating them would create drift.

5. **Add a scoped processing mode instead of replaying all data.**
   - Rationale: the user explicitly wants append-in-the-end behavior for one monthly/corrected file, not a two-million-row historical replay.

6. **Idempotency must be visible.**
   - Rationale: duplicates are expected; result UI and logs must separate inserted rows from already-existing or skipped rows.

7. **UI upload entry point is mutating, but opportunity browsing stays read-only.**
   - Rationale: manual intake is an operator action; Explorer/Radar row/card actions should remain non-mutating except opening the upload flow.

8. **Premium polish remains operational.**
   - Rationale: use stronger hierarchy, refined cards, status chips, subtle surfaces, and document-like detail grouping; avoid decorative marketing hero treatment.

## Proposed Flow

1. User clicks green `Cargar CSV` button in the workspace header.
2. UI opens a modal or right-side sheet with:
   - dataset selector: `Licitaciones` or `Ordenes de compra`.
   - drag-and-drop area.
   - file picker fallback.
   - CSV-only constraints and max-size copy.
3. User selects file.
4. Backend preflight validates:
   - exactly one file.
   - `.csv` extension and expected text/csv-compatible content.
   - size limit.
   - encoding compatibility and parseability without corrupting required header names.
   - parseable semicolon-delimited CSV.
   - required columns for selected dataset.
   - row count and SHA-256.
5. UI shows preflight summary:
   - chosen dataset.
   - original filename.
   - canonical intake filename.
   - row count.
   - missing column errors if any.
   - duplicate-file/hash warning if known.
6. User confirms processing.
7. Backend creates or reuses operational records:
   - `source_files`
   - `pipeline_runs`
   - `pipeline_run_steps`
   - `ingestion_batches`
   - optional manual-intake job state if needed.
8. Backend ingests only the uploaded source file into Raw.
9. Backend runs normalized/Silver build in scoped incremental mode for that dataset and new raw rows/source file.
10. UI displays result:
   - processed rows.
   - accepted rows.
   - inserted delta.
   - duplicate/existing rows.
   - rejected rows.
   - normalized/Silver inserted/updated/skipped counts where available.

## API Shape Draft

Endpoints are draft names for proposal planning:

- `POST /uploads/procurement-csv/preflight`
  - multipart file + `dataset_type`.
  - returns validation status, file token, hash, canonical filename, row count, required-column result.
- `POST /uploads/procurement-csv/{file_token}/process`
  - confirms and starts processing.
  - returns job/run identifiers and initial status.
- `GET /uploads/procurement-csv/jobs/{job_id}`
  - returns status, current step, metrics, and errors.

Security and correctness constraints:
- file token must reference server-side staged file only.
- token expires or is single-use after processing/cancel.
- status endpoints expose only metadata, not raw file contents.
- upload path must be outside the read-only dataset mount and inside an app-controlled writable directory.

## Risks / Trade-offs

- **[Risk] Single-file ingest still triggers broad normalized work.**
  - Mitigation: investigate current incremental state and add source-file or raw-id bounded mode before exposing UI processing.

- **[Risk] Raw dedupe by `source_file_id + raw_row_num` can accept same business rows from a renamed file.**
  - Mitigation: rely on normalized/Silver business-key upserts for canonical dedupe, and add visible metrics that distinguish raw accepted rows from canonical inserted deltas.

- **[Risk] File uploads introduce security exposure.**
  - Mitigation: strict size limits, extension/content checks, sanitized names, no path traversal, no direct execution, short-lived staged files, and operator-only route assumption until auth exists.

- **[Risk] UI polish could blur read-only boundaries.**
  - Mitigation: keep Explorer/Radar actions read-only; only `Cargar CSV` enters a clearly labeled operator upload flow.

- **[Risk] Status polling can overload backend.**
  - Mitigation: bounded polling interval, stop polling on terminal status, and lightweight status payloads.

- Keep full dataset commands unchanged.
- If downstream scoped processing is unsafe, ship preflight/stage only and keep process action disabled until scoped mode is proven.
